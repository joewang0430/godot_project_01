# Godot C# 与计算机工程 (CE) 底层执行时序图

为了以最严谨的底层思维（内存分配、指针、系统栈/堆、CPU周期）来拆解 Godot 引擎的运行机制，我们将任何游戏逻辑的生命周期严格锚定在以下四个绝对的时序阶段（T1 - T4）。

## T1: 引擎启动期（Engine Boot）
* **发生时机**: 你双击 `.exe` 或者在编辑器里按下 F5 的那一瞬间。
* **硬件行为**: 操作系统（OS）拉起进程。C++ 核心代码被装载进只读代码段（Text Segment）。初始化底层的 C++ 物理服务器（PhysicsServer）和渲染服务器（RenderingServer）。启动 .NET 虚拟机 (CLR)。
* **数据状态**: 一切都是空的。没有场景，没有节点。

## T2: 场景反序列化阶段（Load .tscn）
* **发生时机**: `get_tree().change_scene_to_file("...tscn")` 或引擎加载初始场景。
* **硬件行为**: CPU 开始读取 `.tscn` 纯文本文件（其实是个类似 JSON/XML 的树状数据单）。解析器在 **C++ 堆内存 (Heap)** 中疯狂进行 `malloc()` 或 `new`，按照文本里的描述，把 C++ 版本的 Node（比如 CharacterBody2D）一个一个在内存里实例化出来，并连成一棵 N叉树（BVH或普通的层级树）。
* **数据状态**: C++ 物理躯壳存在了，但 C# 的灵魂（你写的脚本）还没有挂载上去。

## T3: 脚本绑定阶段（Script Instantiation & FFI）
* **发生时机**: 场景加载完成的末尾，也就是每个节点的 `_Ready()` 函数被按 DFS（深度优先）遍历触发的瞬间。
* **硬件行为**:
    * C# 虚拟机在 **托管堆 (Managed Heap)** 里分配你写的类（如 `Player`）的内存。
    * 将 C# 对象的指针与 C++ 对象的指针进行**硬绑定**。
    * 执行你写的 `_Ready()`。此时是在做内存“搞地皮”、状态机对象初始化（`new` 操作）和跨语言读取初始配置（比如获取 `Gravity`）。
* **数据状态**: 所有对象（C++ 和 C#）的内存地址固定下来，且虚函数表（vtable）和状态机内部指针互相指向完毕。绝对不再在这个阶段之后做大量的 `new` 操作。

## T4: 游戏主循环（Game Loop & Stack 执行）
* **发生时机**: 游戏正式开始，玩家看到第一帧画面。
* **硬件行为**: 
    * 这是一个死循环：`while (true)`
    * CPU 靠高频计时器，以 60Hz 或无限帧率疯狂触发 FFI 调用。
    * C++ 引擎通过函数指针直接 `CALL` 到你写在 `_PhysicsProcess` 或 `_Process` 里的代码段。
    * **极客铁律**: 在这个阶段，所有的计算必须在栈（Stack）和寄存器（Registers）上高速完成。尽量减少拉动 FFI 的跨界通信，严禁在此阶段 `new` 新对象（以防止堆内存产生碎片和触发 GC），全程依赖 T3 阶段分配好的指针做 `O(1)` 的状态切换。

---
> **阅读本工程代码前声明**：
> 这里的所有系统与架构设计，都必须严格对照上述 T1-T4 进行解释。无论是变量声明、函数调用还是设计模式，都必须指明它发生在哪一个时序，对 CPU 和内存的物理影响是什么。
